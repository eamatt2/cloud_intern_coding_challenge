# Nucleus Security Cloud Intern Project

This repository contains a small web service that runs in Docker and is accessible over HTTPS using a self-signed cert that is generated on startup.

## Architecture

The project has three containers that are managed by Docker Compose.

**Certgen** runs once at startup using the 'alpine/openssl' image. It generates a self-signed TLS certificate and writes it to a shared volume. The other containers wait for this process to finish before they start.

**nginx** is exposed to the host on ports 80 and 443. It handles TLS termination using the cert that was generated by certgen. It then redirects all HTTP traffic to HTTPS. Then it forwards requests to the internal app container over HTTP.

**app** runs the Flask application using gunicorn. It can only be reached within the Docker network, so all traffic must go through nginx.

## Running locally

Docker Desktop is needed on Windows and macOS

### 1 - Start the stack 

After cloning the repo, start the stack by running:
```
docker compose up --build
```

Docker will generate the TLS cert, build the app, and start the services.

### 2 - Test it

Open a browser and go to 'https://localhost'. This will show a pop up warning, but you can proceed to the site by clicking the 'Advanced' button.

You can also test by using curl:
```
curl -k https://localhost/
curl -k https://localhost/admin
```

-k tells curl to accept the self-signed certificate.

Confirm HTTP -> HTTPS redirect is working:
```
curl -v http://localhost/
```

This should return a 301.

### 3 - View logs
```
docker compose logs -f
```

### 4 - Stop the stack
```
docker compose down
```

## Design choices

### Cert Generation In Docker
I decided to use alpine/openssl container because it avoided dependency on the host OS. It works the same on Windows, mac, and Linux. I originally had it run a bash script, but ran into path issues on Windows.

### nginx TLS termination
The web app does not touch the certificates which is similar to how a load balancer handles TLS.

### Flask and gunicorn
Flask is minimal and easy to read and understand. gunicorn replaces Flask's built-in dev server. gunicorn is able to run multiple workers and logs the requests to stdout

### Non-root user
Having a non root user satisfies the principle of least privilege. If the container is compromised it reduces the damage that the attacker can do.

### Expose vs ports for the app
The Flask service uses expose for internal use rather than ports. nginx is the only public facing entry point.

### Docker Compose conditions
The 'condition: service_completed_successfully' condition makes sure that nginx and the app do not start until cert generation has finished

## What I Would Improve
1. I would use mkcert to issue a cert from a locally-trusted CA so that the browser warning does not pop up.
2. I would use Docker secrets or a secret manager that injects the secrets at runtime instead of relying on a volume.
3. I would add a healthcheck in docker compose to improve dependency checks so that app and nginx properly wait for certgen. This would be useful if the app was to be expanded and if more services are added to it.

## Deploying to the cloud with AWS

What separates the cloud from running locally is that TLS is handled by the ACM and the ALB, so we can either remove nginx or simplify it. 

1. Building and pushing to ECR: Push the image to Amazon Elastic Container registry after you build the image and tag it.
2. Create an ECS Task: Reference the ECR image and set CPU and memory limits. Then you can use the AWS secrets manager to inject the environment variables.
3. Create ECS service with ALB: Setup the Application Load Balancer to terminate TLS and use ACM certificates. These certificates are free and will auto renew. Next the ALB will forward the HTTP traffic internally
4. Configure DNS with Route 53: Next we map a domain name to the ALB's DNS name
5. Assign IAM roles: Grant ECS only the permission it needs so that we can ensure principle of least privilege.

## Why storing a private SSL key in a repository is bad
A private key is the second half of a certificate pair. If someone gets access to it, they can impersonate your service and decrypt TLS traffic that is directed at it.

There are a few risks that come with storing it in a repository:
1. Git history can be checked, so even if you delete the file in a commit, the key can still be recovered and used.
2. Some services like github have secret scanners, but those are not always reliable so your key can still be leaked through them.
3. If your repository is public then that is viewable by everyone, that means that automated scanners can grab the SSL key. Even misconfigured private repos can give access to this key to attackers.

To remediate this, keys should be generated outside of the repo and then injected at runtime by using a secret manager or Docker secrets. This improves security as you can configure access rules on your secret manager and do not have to worry about repository leaks.
